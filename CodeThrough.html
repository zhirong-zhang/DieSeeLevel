Die, See, Level

A first-person action game prototype built in Unreal Engine 5.7. You physically rotate a PS5 DualSense controller to tilt an in-game level indicator and change the world's gravity. Touching an egg spawns an enemy — keep flipping gravity to dodge their attacks.

This doc covers the specific problems we ran into building each core system, and what we ended up doing about them. Not a tutorial, not a design doc — just a writeup of the bugs and dead ends so the code makes sense when you come back to it later.

Part 1

Gyro-Driven Gravity System

What It Does Reads gyroscope data from the PS controller to rotate an in-game level indicator (GravityPlane), which drives the player character's gravity direction.
1

UE has no native support for PlayStation controllers

The Problem

UE doesn't recognize PS5 DualSense out of the box — buttons and axes both come back empty.

The Fix

Manually register the controller in Project Settings → Raw Input using its VID/PID, map the raw axis indices (e.g. GenericUSBController Axis 1) to engine input names, then bind them in SetupPlayerInputComponent via Enhanced Input.

Reference: BV1g4HqeiEFF
2

Raw Input can't read gyroscope data from the PS controller

The Problem

Raw Input only handles standard axes and buttons. The gyroscope data is completely outside that system.

The Fix

Switched to the Controller Support plugin, which talks directly to the hardware SDK and registers the live IMU orientation as float vector input channels the engine can read.

Reference: rafaelvaloto/Unreal-Dualsense
3

Assigning raw Euler angles directly causes value jumps and gimbal lock

The Problem

Directly assigning the plugin's absolute Euler angles to the object causes sudden flips when crossing ±180°, and Euler angles have inherent gimbal lock — at certain orientations the rotation becomes completely unpredictable.

The Fix

Switched to per-frame delta accumulation + quaternion composition. Each frame builds a relative offset from the starting quaternion recorded at button press, which sidesteps Euler singularities entirely.

Code — FPSCharacter.cpp · Tick()
FPSCharacter.cpp Tick() · lines 104–128 
// convert gyro angular velocity to a delta angle for this frame
float PitchDelta = -CurrentGyroRotation.Pitch * DeltaTime * GyroGravitySensitivity;
float RollDelta  =  CurrentGyroRotation.Roll  * DeltaTime * GyroGravitySensitivity;

float AbsPitch = FMath::Abs(PitchDelta);
float AbsRoll  = FMath::Abs(RollDelta);

// single-axis lock: only the dominant axis accumulates, suppresses gyro noise
if (AbsPitch > AbsRoll && AbsPitch > 0.01f)
    AccumulatedPitch += PitchDelta;
else if (AbsRoll > AbsPitch && AbsRoll > 0.01f)
    AccumulatedRoll  += RollDelta;

// compose the two axis deltas onto the starting quaternion
// quaternion multiply instead of FRotator addition — no gimbal lock
FQuat QPitch(FVector::RightVector,   FMath::DegreesToRadians(AccumulatedPitch));
FQuat QRoll (FVector::ForwardVector, FMath::DegreesToRadians(AccumulatedRoll));
FQuat NewRelativeRot = InitialPlaneQuat * QPitch * QRoll;

GravityPlane->SetRelativeRotation(NewRelativeRot);
4

RInterpTo step size shrinks near the target — rotation stalls and never finishes

The Problem

After releasing the button, the view gets stuck in the last few degrees. RInterpTo takes smaller and smaller steps as it approaches the target — floating point drift means those final fractions take dozens of frames to clear, and the "done" condition never triggers.

The Fix

Added an early exit: check the quaternion AngularDistance each frame, and hard-snap to the target once it drops below 1°. Don't wait for the interpolation to converge naturally. Write the new gravity direction at the same time.

Code — FPSCharacter.cpp · Tick()
FPSCharacter.cpp Tick() · lines 130–165 
// smooth interpolation each frame
FRotator NewActorRot = FMath::RInterpTo(
    CurrentActorRot, TargetActorRotation, DeltaTime, SmoothRotationSpeed);

SetActorRotation(NewActorRot);
if (Controller)
    Controller->SetControlRotation(NewActorRot);

// quaternion angular distance: stable regardless of Euler representation ambiguity
float AngleDifference = FMath::RadiansToDegrees(
    NewActorRot.Quaternion().AngularDistance(TargetActorRotation.Quaternion()));

// early exit: hard-snap at <1° to kill floating point tail
if (AngleDifference < 1.0f)
{
    SetActorRotation(TargetActorRotation);
    if (Controller)
        Controller->SetControlRotation(TargetActorRotation);

    // Z+ = new up direction, negate for gravity
    FVector SnappedUp = TargetActorRotation.Quaternion().GetAxisZ();
    GetCharacterMovement()->SetGravityDirection(-SnappedUp);

    // reset to Falling so CMC finds the new floor under the new gravity
    GetCharacterMovement()->SetMovementMode(MOVE_Falling);

    bIsSmoothRotating = false;
    SpawnRoomObjects(); // gravity aligned — trigger level refresh
}
5

After a gravity flip, standard movement and camera logic both break

The Problem

Once the character's up vector is no longer world Z, the engine's Controller rotation system still operates in world space — movement goes diagonal, the camera drifts, and at worst the character spins in place and can't turn.

The Fix

Cut the Controller rotation coupling entirely and drive the Actor directly:

  • Disable it in both the constructor and BeginPlay — some UE systems quietly re-enable it during init, so once isn't enough.
  • Movement: use GetActorForwardVector() / GetActorRightVector(). These are local to the Actor and automatically correct after it rotates to the new gravity.
  • Yaw: quaternion rotation around the Actor's local up axis via AddActorLocalRotation — the axis is always right regardless of which surface you're on.
  • Pitch: CurrentCameraPitch accumulates manually and writes directly to the camera's relative rotation, bypassing Controller space completely.

void AFPSCharacter::Look(const FInputActionValue& Value)
{
    if (bIsShiftingGravity || bIsSmoothRotating) return;

    FVector2D Axis = Value.Get<FVector2D>();
    if (Controller != nullptr)
    {
        float YawInput   =  Axis.X * GamepadLookSensitivity * GetWorld()->GetDeltaSeconds();
        float PitchInput = -Axis.Y * GamepadLookSensitivity * GetWorld()->GetDeltaSeconds();

        // Yaw: rotate the whole Actor around its local up axis — correct on any surface
        FQuat YawRotation(FVector::UpVector, FMath::DegreesToRadians(YawInput));
        AddActorLocalRotation(YawRotation);

        // Pitch: accumulate manually and write to camera relative rotation, bypassing Controller space
        CurrentCameraPitch = FMath::Clamp(CurrentCameraPitch + PitchInput, -89.0f, 89.0f);
        FirstPersonCamera->SetRelativeRotation(FRotator(CurrentCameraPitch, 0.f, 0.f));
    }
}
Part 2

Boss Tracking in a Multi-Gravity World

What It Does The Boss stays synced to whatever gravity the player is currently using, maintains correct body orientation, and keeps chasing — even when gravity flips mid-pursuit.
1

AI characters are locked to world gravity and can't follow the player's gravity switch

The Problem

AI only knows about world-absolute gravity. After a flip the Boss just stands there on the old surface, orientation unchanged. On top of that, dynamically spawned AI doesn't get a Controller by default, so it can't move at all.

The Fix

Read the player CMC's gravity direction every frame and compare it against the Boss's own. On mismatch, sync immediately — and do two things at the same time to prevent physics explosions: switch to MOVE_Falling to break off the old surface, and nudge the capsule 60 units upward along the old up vector so it doesn't clip into the new floor mid-rotation. Orientation is handled with quaternion interpolation — Z axis to new up, X axis toward the player. AutoPossessAI in the constructor handles the Controller problem, and AirControl = 1.0 stops the Boss from freezing mid-air right after a switch.

Code — BossAI.cpp · Constructor
BossAI.cpp Constructor · lines 8–28 
// disable Controller rotation coupling — we drive orientation manually with quaternions
bUseControllerRotationPitch = false;
bUseControllerRotationYaw   = false;
bUseControllerRotationRoll  = false;
GetCharacterMovement()->bOrientRotationToMovement = false;

// auto-bind AI Controller whether placed in level or dynamically spawned
AutoPossessAI = EAutoPossessAI::PlacedInWorldOrSpawned;

// after a gravity switch the Boss may be mid-air — AirControl=1.0 keeps it responsive
GetCharacterMovement()->AirControl = 1.0f;
Code — BossAI.cpp · Tick()
BossAI.cpp Tick() · lines 53–92 
// read from the player's CMC, not the player's UpVector — UpVector wobbles during rotation transition
FVector TargetGravity  = PlayerCMC->GetGravityDirection();
FVector TargetUpVector = -TargetGravity;

FVector BossCurrentGravity = GetCharacterMovement()->GetGravityDirection();

if (!BossCurrentGravity.Equals(TargetGravity, 0.01f))
{
    GetCharacterMovement()->SetGravityDirection(TargetGravity);

    // anti-explosion ①: break off the old floor's surface snapping
    GetCharacterMovement()->SetMovementMode(MOVE_Falling);

    // anti-explosion ②: nudge capsule upward so it doesn't clip into the new floor
    AddActorWorldOffset(GetActorUpVector() * 60.0f, false);
}

// build target orientation: Z = new up, X = toward player
FVector ToPlayer      = TargetPlayer->GetActorLocation() - GetActorLocation();
FVector MoveDirection = FVector::VectorPlaneProject(ToPlayer, TargetUpVector).GetSafeNormal();

if (!MoveDirection.IsNearlyZero())
{
    FQuat TargetQuat = FRotationMatrix::MakeFromZX(TargetUpVector, MoveDirection).ToQuat();

    // quaternion slerp: no gimbal lock, shortest rotation path
    FQuat NewQuat = FMath::QInterpTo(GetActorQuat(), TargetQuat, DeltaTime, RotationSpeed);
    SetActorRotation(NewQuat);
}
2

NavMesh completely breaks in a multi-gravity environment

The Problem

NavMesh is baked against a standard Z-up floor. Once the player is on a wall or ceiling there's no valid path, so the Boss just stops. Rebaking dynamically is theoretically possible but way too slow to run on every gravity switch.

The Fix

Skip NavMesh entirely. Each frame compute the vector from Boss to player, project it onto the current gravity plane, and push with AddMovementInput. Movement is blocked until orientation is close enough to the new gravity (angle > 2° = still rotating, don't chase yet).

Code — BossAI.cpp · Tick()
BossAI.cpp Tick() · lines 76–104 
// check how far the Boss's up vector is from the target up vector
float AngleDiff = FMath::RadiansToDegrees(
    FMath::Acos(FVector::DotProduct(GetActorUpVector(), TargetUpVector)));
bool bIsAligningGravity = (AngleDiff > 2.0f);

// pure vector pathfinding: project toward-player direction onto gravity plane
FVector ToPlayer      = TargetPlayer->GetActorLocation() - GetActorLocation();
FVector MoveDirection = FVector::VectorPlaneProject(ToPlayer, TargetUpVector).GetSafeNormal();

// don't move until orientation is settled — prevents chasing sideways mid-flip
if (!bIsAligningGravity)
{
    float DistanceToPlayer = ToPlayer.Length();
    if (DistanceToPlayer > StopDistance)
    {
        // direct AddMovementInput — no NavMesh involved
        AddMovementInput(MoveDirection, 1.0f);
    }
}
Part 3

DynamicGravityComponent — Physics Prop Gravity Sync

What It Does A reusable component you attach to any physics-simulated Actor. Makes scene props (crates, etc.) move under whatever gravity direction the player is currently using, so the physical world stays consistent after every flip.
1

Physics props have no way to follow the player's gravity direction

The Problem

Chaos Physics uses a global fixed gravity. After the player flips gravity, every prop in the scene still falls toward world Z-down.

The Fix

In BeginPlay, disable the mesh's built-in gravity and cache a reference to the player. Each tick, read the player CMC's gravity direction and manually apply the equivalent acceleration to the prop, taking full ownership of its gravity behavior.

Code — DynamicGravityComponent.cpp · BeginPlay()
DynamicGravityComponent.cpp BeginPlay() · lines 12–27 
// get the root component of whichever Actor this is attached to (the physics mesh)
PhysicsComp = Cast<UPrimitiveComponent>(GetOwner()->GetRootComponent());

if (PhysicsComp && PhysicsComp->IsSimulatingPhysics())
{
    // kill built-in gravity — this component takes over from here
    PhysicsComp->SetEnableGravity(false);
}

// cache player reference so we can read gravity direction each tick
PlayerCharacter = UGameplayStatics::GetPlayerCharacter(GetWorld(), 0);
2

Rigid body sleep causes props to teleport on gravity switch

The Problem

Chaos marks idle rigid bodies as sleeping. Sleeping bodies drop applied forces on the floor. So when gravity switches, the new force gets ignored — the prop just sits there until something bumps it, then snaps suddenly. Looks like a teleport.

The Fix

Compare gravity direction every frame. The moment it changes, call WakeRigidBody() before applying the force.

Code — DynamicGravityComponent.cpp · TickComponent()
DynamicGravityComponent.cpp TickComponent() · lines 35–43 
FVector CurrentGravityDir = PlayerCharacter->GetCharacterMovement()->GetGravityDirection();

// detect gravity direction change and wake the body before applying force
if (!CurrentGravityDir.Equals(LastGravityDir, 0.01f))
{
    PhysicsComp->WakeRigidBody();  // force out of sleep so the next AddForce actually lands
    LastGravityDir = CurrentGravityDir;
}
3

Different-mass props fall at different rates when force is applied manually

The Problem

AddForce applies a force, not an acceleration — a light prop goes flying while a heavy one barely moves. Every prop ends up falling at a different speed.

The Fix

Pass true as the third argument to AddForce (bAccelChange). That switches it to acceleration mode — the engine multiplies by mass internally, so every prop falls at exactly 980 cm/s² regardless of weight.

Code — DynamicGravityComponent.cpp · TickComponent()
DynamicGravityComponent.cpp TickComponent() · lines 45–49 
// bAccelChange = true: pass acceleration, not force
// engine multiplies by mass internally — all props fall at 980 cm/s² regardless of weight
PhysicsComp->AddForce(CurrentGravityDir * 980.0f, NAME_None, true);
Part 4

Boss Spawn System

What It Does The moment gravity alignment completes, VentSpawners get placed on random walls and immediately eject a Boss into the room — refreshing the level after every gravity flip.
1

Catching the exact moment gravity alignment finishes

The Problem

During the rotation transition the coordinate system is unstable — raycasts fired mid-rotation come out at wrong angles. But using a Timer or Blueprint notify makes it hard to guarantee you're actually synced with the physics state.

The Fix

Call SpawnRoomObjects() as the very last line of the angle convergence block in Tick. An angle diff < 1° means alignment is done and the coordinate system is trustworthy. bIsSmoothRotating gets set to false first to ensure it only fires once per flip.

Code — FPSCharacter.cpp · Tick()
FPSCharacter.cpp Tick() · lines 141–165 
float AngleDifference = FMath::RadiansToDegrees(
    NewActorRot.Quaternion().AngularDistance(TargetActorRotation.Quaternion()));

if (AngleDifference < 1.0f) // aligned — coordinate system is stable
{
    SetActorRotation(TargetActorRotation);
    if (Controller)
        Controller->SetControlRotation(TargetActorRotation);

    FVector SnappedUp = TargetActorRotation.Quaternion().GetAxisZ();
    GetCharacterMovement()->SetGravityDirection(-SnappedUp);
    GetCharacterMovement()->SetMovementMode(MOVE_Falling);

    bIsSmoothRotating = false; // set false first so SpawnRoomObjects can't re-trigger

    // coordinate system is stable — spawn now
    SpawnRoomObjects();
}
2

Filtering random raycasts for valid wall spawn points

The Problem

Random rays hit floors, props, and other surfaces that shouldn't get vents — spawning on those breaks things. But you can't just retry infinitely or the frame stalls.

The Fix

Cap retries at 10. On each hit, check whether the Actor has the SpawnableWall tag — no tag, move on; all 10 miss, give up and skip this spawn. Level designers just tag the surfaces where vents are allowed to appear.

Code — FPSCharacter.cpp · TryFindRandomWallSpawnPoint()
FPSCharacter.cpp TryFindRandomWallSpawnPoint() · lines 297–349 
for (int32 i = 0; i < 10; ++i) // max 10 attempts — don't stall the frame
{
    FVector RandomDirection = FMath::VRand(); // uniform random direction on a sphere
    FVector EndLocation = StartLocation + RandomDirection * 3000.0f;

    FHitResult HitResult;
    FCollisionQueryParams QueryParams;
    QueryParams.AddIgnoredActor(this);

    if (GetWorld()->LineTraceSingleByChannel(
            HitResult, StartLocation, EndLocation, ECC_Visibility, QueryParams))
    {
        AActor* HitActor = HitResult.GetActor();

        // only surfaces tagged SpawnableWall are valid spawn points
        if (HitActor && HitActor->ActorHasTag(FName("SpawnableWall")))
        {
            OutLocation = HitResult.ImpactPoint;
            OutRotation = FRotationMatrix::MakeFromX(HitResult.ImpactNormal).Rotator();
            return true;
        }
    }
}
return false; // all 10 missed a valid wall
3

Vent auto-facing outward, Boss spawn position avoiding wall geometry

The Problem

The vent needs to face outward correctly on any surface — side walls, ceiling, whatever the new floor is after a flip. If the Boss spawns directly at the hit point, the capsule clips into the wall and gets launched by the physics engine.

The Fix

Build the vent's rotation from the raycast's ImpactNormal as the X axis (MakeFromX). The surface normal is always pointing outward, so the vent faces the room automatically on any surface — no manual rotation needed. VentSpawner's SpawnArea is offset 100 units along that X axis, so the Boss spawns in open air in front of the wall instead of inside it.
